# ADR-001 백엔드 프레임워크 선택

## 1. 배경
- 이 프로젝트는 서버사이드 렌더링이 아닌, REST API 중심의 백엔드 서비스를 목표로 한다.
- 팀 내 백엔드 담당 인원은 3명이다.
- 해당 팀원 3명 모두 부트캠프에서 Java + Spring 기반 백엔드 과정을 수료했다.

## 2. 당시 상황
- 프로젝트 시작 시점에는 일정 여유가 크지 않았고, 새로운 언어나 프레임워크를 학습할 시간보다는
  **기능 구현 및 협업 방식 정착**에 시간을 쓰는 것이 중요했다.
- 따라서, 팀이 이미 익숙한 기술 스택을 사용하는 것이 리스크가 낮다고 판단했다.

## 3. 결정
- 백엔드 프레임워크로 **Spring Boot**를 사용한다.

## 4. 선택 근거
- **팀 역량**
    - 백엔드 담당 3명 모두 Java + Spring 실습 경험이 있다.
    - 부트캠프에서 이미 스프링 기반으로 미션/프로젝트를 수행해본 경험이 있어, 기본적인 사용법을 알고 있다.
- **러닝 커브**
    - 새로운 프레임워크(예: Node.js, NestJS, Django 등)를 도입할 경우, 학습 부담이 커지고 프로젝트 일정에 영향이 클 것으로 예상되었다.
    - Spring Boot는 이미 익숙한 만큼, "언제까지 무엇을 만들 수 있는지"를 대략적으로 예측할 수 있었다.
- **생태계**
    - Java + Spring Boot는 백엔드에서 많이 쓰이는 조합이며, 레퍼런스/질문/자료를 찾기 쉽다.
    - (실제 프로젝트 수행 중에도, 검색을 통해 문제 해결이 쉬웠다.)

## 5. 영향
- **긍정적**
    - 초기 생산성이 높았다. 팀원 간에 "이건 어떻게 구현하지?"보다는 "어떻게 구조를 나눌까?"에 더 집중할 수 있었다.
    - 기존 부트캠프 경험을 재활용할 수 있어서, 안정적으로 기본 기능을 구현할 수 있었다.
- **제약**
    - Java/Spring에 익숙하지 않은 사람이 새로 합류할 경우 진입장벽이 있을 수 있다.
    - "다른 언어/프레임워크를 실험해보는 경험"은 이번 프로젝트에서는 우선순위에서 밀렸다.

## 6. 향후 논의 가능 포인트
- 차기/다음 프로젝트에서는 팀 역량에 여유가 생긴다면, 다른 백엔드 프레임워크(Node/Nest/Django 등)를 비교 검토해볼 수 있다.
