# 협업 규칙

## 1. 목적
- 이 프로젝트에서 함께 작업할 때 지켜야 할 기본 협업 규칙을 정리합니다.
- 새로 합류한 사람이 이 문서만 읽어도 Git / PR / 코드리뷰 방식 등을 이해할 수 있도록 합니다.

## 2. Git 브랜치 전략
- 기본 브랜치:
    - `main`: 배포 가능한 안정 버전
    - `develop`: 기능 개발 통합 브랜치 (필요 시)
- 기능 브랜치 예시:
    - `feature/이슈번호-짧은-설명`
- 머지 원칙:
    - `feature/*` -> `develop` PR 후 리뷰 최소 1명 승인
    - `develop` -> `main` 은 스프린트/릴리즈 단위로 머지

## 3. 커밋 메시지 규칙
- 포맷: `<type>: <내용>`
- type 예시:
    - `feat`: 기능 추가
    - `fix`: 버그 수정
    - `chore`: 빌드/설정/기타
    - `docs`: 문서 수정
- 예:
    - `feat: 로그인 실패 시 에러 메시지 표시`
    - `fix: 회원가입 시 닉네임 중복 검사 수정`

## 4. PR 규칙
- PR 제목: `[타입] 작업 요약`
    - 예: `[feat] 게시글 목록 조회 API 추가`
- PR 내용에 포함:
    - 작업 내용 요약
    - 테스트 여부
    - 관련 이슈 링크
- 리뷰 원칙:
    - 최소 1명 이상 승인 후 머지
    - 자기 PR은 자신이 머지하지 않는다(팀 상황에 맞게 조정)

## 5. 이슈 / 보드 사용
- 이슈 제목 규칙:
    - `[#도메인] 한 줄 설명`
    - 예: `[회원] 로그인 실패시 500 에러`
- 라벨 예시:
    - `bug`, `feature`, `chore`, `question`
- 작업 플로우 예:
    - 이슈 생성 → 브랜치 생성 → 작업 → PR → 리뷰 → 머지 → 이슈 닫기
